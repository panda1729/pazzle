# ADR-0004: 行き止まり破壊(braid)によるループ導入でルート選択を作る

- Status: Accepted
- Date: 2026-07-19

## Context

「面白くない」というユーザーフィードバックの根本原因を分析したところ、次の3点に集約された。

1. チェックポイントの手前に崩れる床を置く、というワンパターンなギミック配置になっている
2. ×2マス(heavy)が正解ルート上にしか置かれておらず、「踏むか迂回するか」の選択が生まれない
3. 迷路が基本一本道で、難易度が「長さ」でしか上がっていない

`generateMaze`(再帰バックトラック法)が返す迷路は木構造、つまり任意の2点間の経路が常に一意な**完全迷路**である。経路が一意である以上、その上にどんなギミックを置いても「別ルートを選ぶ」という意思決定は原理的に発生しない。crumble を CP 手前に置く設計に偏っていたのも、経路が1本しかない中で「無駄な往復をさせる」以外にギミックを機能させる手段がなかったためであり、症状ではなく完全迷路という構造そのものが原因だった。

## Decision

- `generateMaze(size, seed, braid = 0)` に `braid`(0〜1)パラメータを追加する。迷路生成後、行き止まりマス(開通方向が1つだけのマス)を走査し、確率 `braid` で既存の開通方向以外の壁を1枚(既存 rng で決定的に)壊してループ(閉路)を作る。`braid = 0` は従来の完全迷路と完全に同一(後方互換)。
- `StageDef.braid` / `Stage.braid` を追加し、`buildStage` が `generateMaze` に渡す。既存の `heavyCells` 等と同じ「省略時は0に正規化」パターンに合わせた。
- ループができたことで生まれる「短いが不安定/条件付きの近道」と「長いが確実な迂回路」の差を使って、STAGE 04/05/07/08 を次の原則で再設計した。
  - **HEAVY(×2マス)**: 最短ルートは踏むが、少し遠回りすれば避けられる位置に置く。全て迂回した場合のコストが `par+1`〜`par+6` に収まることを定量条件にした(踏むのが最適だが、迂回も比較検討に値する範囲)。
  - **CRUMBLE(崩れる床)**: 「CPの手前に置く」パターンをやめ、ループが作る一度きりの近道として `uses: 1` で配置する。全チェックポイントからチェビシェフ距離2以上離すことで、往復パターンの再発を構造的に防いだ。塞いだ場合のコストが `par+2` 以上になることを定量条件にした。
  - **WARP(STAGE07/08)**: 使う場合と使わない場合のコスト差が3以内になる位置に置き、「得か損か一見わからない」選択にした。
- これらの定量条件は `stages.test.ts` に設計ガードとして追加し、将来ステージを調整してもパターンの再発が自動的に検出されるようにした。

## Consequences

- ルート選択という前提が成立したことで、ギミックを「避けるか踏むか」「使うか温存するか」という意思決定として設計できるようになった。一方で、ギミック同士は互いに最適ルートを変化させうるため(例: ワープを足すと heavy 区間を経由しない別ルートが最適になり、heavy や crumble の「選択性」が消えることがあった)、ステージ設計時は最終形の grid で毎回ソルバーを再実行して定量条件を検証する必要がある。単純に「良さそうな配置を思いつきで置く」だけでは成立しない。
- `braid` が高いほど探索空間が広がり、`findRouteThrough` の計算コストも増える(チェックポイント順列 × Dijkstra)。現状のステージ規模(最大9x9、CP3個)では実用上問題ないが、将来さらに大きい・CPが多いステージを作る場合は計算コストを再検証する必要がある。
- 迷路生成のテスト(`maze.test.ts`)を「木構造であること」に依存させないよう、閉路の存在は辺の本数(`size*size-1` を超えるか)で検証する形にした。
